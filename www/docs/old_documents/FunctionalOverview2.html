  1. Executive Overview

To: Political firebrands. Cultural Creatives. Muckrakers. Small businesses.

Subject: You Deserve a Voice

Message:


The Writeable Web is a free non-commercial browser that turns the inmates loose to run the asylum by radically including everything necessary to author and view content within the Writeable Web application itself. This transforms web sites from static information silos to spaces where members can collaborate and communicate using the editing tools built right into the browser. Writeable Web includes wizards and templates for creating weblog diaries, online political newspapers, and more.


The Old Web stores web sites on centralized servers that are expensive and difficult to maintain; Writeable Web sledgehammers Writeable Web Sites, smashing them into hundreds of fragments that are securely stored across a multitude of machines. When the Writeable Web Site is requested, the fragments coalesce like drops of mercury in a process known as Peer to Peer similar to the music-sharing program Napster. This peer to peer process makes it possible to create new Writeable Web Sites for free. Instead of pulling out a credit-card and configuring a web-server, you simply save your pages just as you save documents in your word processor today. No fuss no muss.


Writeable Web also colorizes the monotone domain name system. Your world isn't just about .COMpanies, .EDUcational institutions, or .ORGanizations, so why should the web? Writeable Web adds hundreds of free new domain endings, such as .religion, .fan, .sucks, .love, .cooperative, .gay, and more. Plus, you can create your own ending without asking anyone's permission.


And why should the Old Web be left out in the cold? We bring it in and offer it shelter. The Writeable Web automatically add discussion forums across every web page. Further, we empower everyone to easily add links to related web sites for a given site. For example, you can add links pointing users browsing Nike.com to related sites about Nike's labor policy. Finally, the technical underwear of the Writeable Web is designed in such a way that The Old Web can easily put it on.






  1. Executive Walkthrough & Screenshots

This section provides an overview of the major new features The Writeable Web delivers. Features are broken into three major sections: Writeable Web Sites, Next-Generation Browsing, and Seamless Collaboration.


    1. Writeable Web Sites

The engine underlying the features in Writeable Web is a new kind of organic, grassroots web site, named a Writeable Web Site. These new sites are advanced, online communities that completely integrate the tools necessary to create, maintain, and share content. They provide a host of abilities far beyond older web-sites: single-sign on, powerful standardized navigation, integration of all tools needed for editing and creating content in the browser, site members and member roles, instant messaging and group chatting between site members, powerful site and page metadata, pages saved in XML rather than HTML, discussion areas on every page, and the ability for viewers to easily add sites related to the current site.


For more in-depth information on Writeable Web Sites see What is a Writeable Web Site?


    1. Next-Generation Browsing



      1. Single Sign-on and Authenticated Identity

Never again will you have to juggle passwords between Writeable Web Sites. Instead of logging individually into each web site, you log into the Writeable Web browser itself when you first start:


Screenshot of Login window with username BradGNUberg and some password entered.


As you access each Writeable Web Site, an authentication process occurs that involves sending special keys instead of passwords. This happens transparently to the user.


The usernames that are authenticated are known as pseudo-anonymous identities. Writeable Web can always ensure that if a user claims they are HappyGoLucky, for example, that they truly are that user and not someone spoofing that user. However, notice that the username says nothing about who the user really is; instead it simply ensures that some username is truly unique and that other users can't fake being that user. This system has two strong benefits: it ensures responsibility for users on sites by binding them to a certain identity while also allowing them to not reveal their true names.


Writeable Web also makes it easy to have multiple identities, if you are a member of several online communities that may not mesh:


Screenshot of pull-down on login window to select username


Writeable Web also allows you to log into sites as truly anonymous, which means you have no identity. Individual Writeable Web Sites can control whether they allow anonymous users or not.


For more in-depth information on Logging In see Logging In and Out

      1. Using the Index

Writeable Web Sites all support a powerful navigation tool known as the Index:


Screenshot of Index tab on a filled out web site


Studies have shown that user's don't think hierarchically in terms of folders and files, as web-sites are organized today. Instead, they think in terms of concepts and groupings. One of the best existing tools we have for conceptual grouping are book indexes. These indexes collect categories and entries together in an easy-to-find way:


Screenshot of a picture of the index page of a book index, with an arrow to the top-level portions saying “Categories” and arrows pointing to multiple entries saying “Entries”.


Writeable Web Sites are organized similar to book indexes. The web-site is completely “flat”, with no folders. Instead, pages are placed into categories, just like a book index:


Screenshot of a simple Writeable Web index with one category and two entries, with an arrow to the category saying “Category” and two arrows to the entries saying “Entries”


Pages can be in several categories at once, unlike folders. For example, imagine that your company has a business proposal named “Doing Business in Bangkok” that was created by the Sales department. In the past you might have stored this into a single folder on a web-site or your desktop named “Sales Department”. Life isn't so hierarchical, though, and Writeable Web is flexible enough to know this. On a Writeable Web Site you could store “Doing Business in Bangkok” into the categories Sales Department, Thailand, New Business, Items to Keep Track Of, and South-East Asia:


Screenshot showing all of the above categories, with maybe a few more department categories and geographic ones, with the entry talked about above


As your business and needs evolve your categories can evolve and shift with them. Renaming categories, creating new ones, and moving entries is as easy as a click of the mouse.

A Writeable Web index is even better than a book index, though, because the Writeable Web Sites themselves help create parts of the index. Every Writeable Web Site automatically creates categories for members, by page titles, by page type, and for templates:


Screenshot showing the four auto-categories, with arrows comming off them. For members, it says “Easily find information on the members of a site using the Members category”. For Page Titles it says “Easily find pages simply by their titles rather than their categories in the Page Titles category”. For page type categories, two arrows will go to a Weblogs category and a Business Proposals category, saying “Categories are automatically created by the type of template that was used to create a page. This means you can easily see all pages that were created with the Business Proposal or Weblog templates, for example.” For the Templates category, it will say “All page templates that are available for member's to use are in this category. Simply delete entries in this category to keep user's from creating Weblogs, for example, or assign any Writeable Web page into this category for it to instantly become a template for user's to use.”

  1. Introduction

The World Wide Web has become a stagnant play-thing of corporations and inaccessible international organizations. Its original promise of bringing democratic, grassroots speech from everyone to everyone has fallen by the way-side. Now is not the time to give up, dismissing the web as an idealistic dream. Instead, we can create a new evocative vision, a new web that rests on democratic principles.

The Writeable Web is a non-commercial, non-profit project to do exactly this. We are creating a new space, named the Writeable Web, which envelops the existing World Wide Web and moves it forward in innovative, progressive directions. We are replacing and collapsing existing web browsers, web servers, web editors, and office productivity suites into a single, unified new application. In this new web everyone has a voice, regardless of wealth, power, or technical skill, and people can collaborate to create spontaneous, organic spaces for politics, business, and fun. It is easy to comment on the powerful, and the web becomes human-centered. It is a profoundly democratic, decentralized vision.

The 11 Principles of The Writeable Web

The engine of Writeable Web is a next-generation web site known as a Writeable Web Site. At their most basic Writeable Web Sites are simply super-charged web-sites that make it possible to create, maintain, share, and discuss documents accessible only to the people you want. Underlying Writeable Web Sites, however, are eleven principles that are the heart of the Writeable Web. These principles are the constitution of this new web, a set of founding principles that we should continiously judge and evolve the Writeable Web against:

  1. Innovation and creativity must be allowed to flourish

  2. Freedom of speech must include affordable access to freedom of speech.

  3. Web sites should be organic, grassroots, and bottom-up

  4. The web should bloom with discussion

  5. It should be easy to find alternative points of view

  6. Everyone should be able to communicate in complex ways

  7. Names are power - and no one should own that power

  8. Web sites should have standardized user-interfaces for navigation and provide more navigation tools

  9. Users should control their information, not companies, governments, or organizations

  10. The web should be smart enough to be a database

  11. Everything should be wrapped into one tool but accessible by other tools

  12. The older World Wide Web can become the Writeable Web if it wants to



Innovation and creativity must be allowed to flourish

Never again should one organization be allowed to control and restrict innovation on the web. Never again should corporate-influenced or fear-induced legislation be allowed to restrict the continued enrichment of the Internet. The Writeable Web must remain open to innovative new directions, and should continiously guard against single points of control, including itself. Let a million crazy-eyed tinkerers attack the Writeable Web.



Freedom of speech must include affordable access to freedom of speech.

Imagine living in a kingdom that gave everyone the right to criticize the royalty without fear of arrest or punishment. Also imagine, though, that in order to open your mouth to make your criticisms you had to pay thousands or hundreds of thousands of dollars. Would you say this society truly has freedom of speech?

This is the situation in most countries today that claim they have free speech. To reach thousands of people by radio, you have to purchase a hundred thousand U.S. dollar radio license. To reach an entire community, you have to own a million dollar newspaper printing press. The only way to communicate is to become a business; sorry, a rich, big business.

The web was supposed to change this and make it possible to reach millions of people even if you aren't a company. The reality is different. One of the tragedies of the web is the Tragedy of the Dot Commons: non-commercial web-sites forced to commercialize to communicate with millions of people. The process goes like this; imagine you are a hobbyist poet who creates art after you come home from work; you share your material with people on your web-site out of love, not out of money. At first your web-site either doesn't cost anything or costs maybe ten dollars a month. However, if you are successful, and people begin to flock to your web-site, you find yourself paying literally thousands of U.S. dollars a month for powerful servers, hosting, and fast server connections. As an artist of course you want to reach millions of people, so you change your .org to a .com and attempt to become a business. Unfortunately, we've learned that the Internet is not the best place to create businesses, so you go out of business, and your art disappears from cyberspace. Blip, your art is now gone from society.

Writeable Web changes this. Instead of having a web site stored on a single centralized server that takes money and technical knowledge to run, Writeable Web Sites are broken into pieces and redundantly stored on each member's or visitor's personal computer. When a user types in a Writeable Web Site address into the Writeable Web program, Writeable Web spontaneously re-assembles the web site and gives it to the user transparently; now this user is himself free to hand out pieces of the Writeable Web Site to those who request it. The upshot of this is that the more user's who access a given web site, the more it is redundantly spread across members' computers, and the better its performance and accessibility becomes. This type of system is known as peer-to-peer, and is similar to the way in which music-sharing programs Napster, Gnutella, and AudoGalaxy work. The fundamental effect of this is that no intervening corporation needs to be paid to run a server or buy expensive server bandwidth from, allowing non-commercial sites to potentially reach thousands of people without the Tragedy of the Dot Commons from occurring.



Web sites should be organic, grassroots, and bottom-up

Writeable Web includes everything needed to create sophisticated, grassroots Writeable Web Sites. First, Writeable Web includes wizards and templates to get you started creating a diverse range of Writeable Web Sites, from weblog diaries to online political newspapers, all through your Writeable Web browser. Writeable Web even includes a powerful, full-featured word processing application that rivals Microsoft Word. Second, membership in your new Writeable Web Site can be restricted to individual friends, family-members, persons in your company, or extended to the entire world. Finally, every page on your new site can be edited right through the Writeable Web browser if the member has the necessary privileges.

This functionality opens a world of opportunities. For example, several people can collaboratively create an online Writeable Web Site investigative journalism newspaper and post articles all completely through the Writeable Web browser. A company could create a Writeable Web intranet complete with company-wide employee weblogs, employee created subprojects, or employee written newspapers, all without top-down authority. This feature, coupled with the fact that Writeable Web Sites are distributed across many members' computers and not stored on a server that suffers from scarcity, should make many innovative, bottom-up collaborative web-sites possible.



The web should bloom with discussion

In the Writeable Web browser is a single tab, titled Comments About This Page, that holds a great deal of power. Every web page, including older World Wide Web pages, automatically have a discussion board attached, no questions asked. For example, go to www.nike.com/MissionStatement.html using Writeable Web, click the Comment tab, and a pane pops open with a discussion area on it for that page.

Writeable Web strikes a balance between the complete chaos of allowing everyone in the world to comment on a given page and the complete control of only allowing members to comment -- it gives everyone the best of both worlds. Users can choose to view only comments posted by the members of that Writeable Web Site or can choose to see non-members comments as well. Since older World Wide Web sites don't have 'members', all comments are posted by non-members and are bunched together.



It should be easy to find alternative points of view

Imagine another fantasy society, similar to the one claiming free speech, where everyone has the right to criticize the powerful. However, imagine that those criticisms have to be inscribed on a needle, and the needle is thrown into a haystack. Voices affirming this society and the powerful, though, get to have their messages written on giant billboards that can't fail to be seen. If you can't easily or readily find dissenting voices in a society, does that civilization really offer freedom of speech?

This society isn't simply imaginary; its how our current world operates. The powerful own the radio stations and the billboards lining our highways; it's easy for us to find their message. What about the dissenters, Socratic voices on the fringes? It's hard to find their messages and alternatives, almost as difficult as finding a needle in a haystack.

Writeable Web changes this. Every site, including older World Wide Web sites, have a What's Related tab that provides a wide range of related sites grouped by different subjects. Some examples of subjects are Labor Policy, Environmental Record, Agree, Disagree, etc. User's can automatically add new subjects and new related sites under each subject without asking anyone's permission. For example, a user could add the subject Pro Life to www.NationalOrganizationOfWomen.org and add the entry www.christiancoalition.org/AlternativesToAbortion.html. Now, whenever user's are browsing www.NationalOrganizationOfWomen.org, they could choose to see links under the Pro Life section for dissenting views on NOW's stance on abortion. Or how about adding the subject Pro Choice to www.christiancoalition.org pointing to www.NationalOrganizationOfWomen.org/KeepAbortionLegal.html? Writeable Web empowers user's to search, find, and provide alternatives to the views represented by the sites they visit, and to share this information with the millions of people frequenting a site.



Everyone should be able to communicate in complex ways

Sure, nowadays anyone can create a simple home-page or a basic collection of web pages. However, can they create a complex online newspaper or a rolling updated weblog diary, for example? Not unless they either have strong technical skills or the money to afford sophisticated server-side content management systems. Everything discussed so far make it possible for ordinary, Average Joes and Janes to create extremely powerful online communities with a few mouse-clicks.



Names are power - and no one should own that power

Domain names, such as www.acme.com or www.myschool.edu, are part of a semi-centralized network known as the Domain Name System (DNS). DNS is currently controlled by an unaccountable, undemocratic organization that has tended to kowtow to large organizations and companies, and which has maintained an artificial scarcity in domain names as bad as the radio spectrum. Writeable Web banishes these bastards (we did a study and decided they are no longer needed) by completely decentralizing the DNS system in a peer-to-peer fashion in exactly the same way as Writeable Web Sites are broken into pieces. Just as making Writeable Web Sites peer to peer makes them createable for free, breaking the DNS system into pieces shared amongst those who use it make it possible to create domain names for free. No longer will you have to pull out your credit card, register a domain name in a difficult process, and wait a few days until it activates. Instead, naming your Writeable Web Site becomes as easy and transparent as naming a file on your local hard-drive; you don't even know it's an issue.

Domain names can also now have spaces in them, and Writeable Web further explodes the DNS name space with hundreds of new domain endings. The venerable .com, .edu, and .org endings are now joined by .religion, .fan, .sucks, .cooperative, .government, .religion.falun gong, .politics.conservative, .country.america, .religion.christian, .goddess, .love, .cooperative, and more. Your world isn't just about .COMpanies, .EDUcational institutions, or .ORGanizations, so why should the web?



Web sites should have standardized user-interfaces for navigation and provide more navigation tools

Application programs, such as Adobe Photoshop or Macromedia Dreamweaver, currently have standardized user-interfaces. They all have pull-down menus, most have toolbars, and most follow similar conventions. This allows new users to quickly switch between different programs and quickly learn how to use them. Web sites, though, have a confusing disarray of user-interfaces. Menus are in completely different areas on different web-sites, if they exist at all. Some are on the left, some are on top, some are in frames, and some are even fun nifty Java applets! While novelty is the spice of life, having to learn a different user-interface for every web-site is a pain.

Writeable Web standardizes the navigation user-interface across all Writeable Web Sites, and introduces a substantial number of new ones. First, all Writeable Web Sites now have an Index panel on the left of their screen which allows users to navigate a Writeable Web Site in the same way they can turn to the index in the back of a book Third, the venerable Back and Forward buttons in browsers gains a new friend, a Front button, which will take you to the front page of a Writeable Web Site. Fourth, all Writeable Web Sites have a standard search tab and embedded search functionality that is always in the same place across all Writeable Web Sites, so users can easily use Search to find information on that site. Fifth, all Writeable Web Sites have a What's New tab that will show user's what has recently changed on that site so user's can quickly come up to speed. Sixth, every Writeable Web Site has a What's Related tab that provides users with directions for finding further information about the current site. And finally, all Writeable Web Sites have a Members tab so user's can easily find out the members of a Writeable Web Site and communicate with them. These features should make it significantly easier for users to do what they do most on the web: find the information they are looking for, without having to relearn this for every web-site.



Users should control their information, not companies, governments, or organizations

Industry players, from Sun to Microsoft, are currently bandying around ideas to store your personal information on centralized servers controlled by them or other unaccountable organizations. While this will make it possible to create single sign-on services for web-sites, it will be at the expense of bolstering current monopolies, losing your privacy, and losing control over your data.

Writeable Web makes single sign-on between Writeable Web Sites possible without giving up control of your personal information. Instead of storing your personal data on a foreign server controlled by someone else, all your information is stored on your local computer. Further, single sign-on merely associates a pseudonym, such as BilboBagins, with a special public and private key. No one else can impersonate BilboBagins, but at the same time no one can truly know that Brad Neuberg is BilboBagins, since this information is not recorded. If you don't even want a pseudonym, you can log into the Writeable Web anonymously. However it should be noted that Writeable Web does not mask or proxy your IP address, hide what you are viewing, or encrypt your communications; it simply logs you into the Writeable Web either as a pseudonym or with no pseudonym at all.



The web should be smart enough to be a database

Writeable Web includes a powerful feature known as AutoTemplates, which are a beginning implementation of Tim Berner Lee's Semantic Web. Writeable Web comes bundled with a wide variety of pre-created templates for your Writeable Web Site for political, non-commercial, and just plain fun communication. One of the bonuses of having users create their Writeable Web Sites through pre-created templates is that the browser can now record exact metadata on what everything actually is so that computers can understand the information. For example, when you create an online newspaper on your web site, Writeable Web knows what are newspaper articles, what are bylines, what are headlines, etc.

AutoTemplates are special templates that allow the browser to save information about every template field in a page as metadata. Since Writeable Web comes bundled with a large number of templates, such as News Articles, To Do Lists, etc., what these fields mean can be standardized across pages. For example, AutoTemplates always know what fields are addresses or customer names, across every template; it knows that a particular person authored a page, or that someone parodied a page. In this way knowledge across the entire Writeable Web is represented in the same way and can be accessed by intelligent computers. Writeable Web turns the web into a giant intelligent database, taking the web to the next level.



Everything should be wrapped into one tool but accessible by other tools

As you can see from the dreams above, Writeable Web collapses what used to be many different systems into a single application. Gone are web servers, stand-alone web browsers, content management systems, web editors, DNS servers, authentication servers, and office productivity suites. The Writeable Web is more accessible because ordinary users now don't need to learn a multiplicity of programs or configure complicated servers.

Even though Writeable Web is an integrated program, it is still open to other developers creating tools that interoperate with it. First, it is completely open-source, so other users can take its code-base and graft on new additions. Further, Writeable Web Sites are saved in an open XML file-format, replacing HTML and proprietary Microsoft Word files, so other programmers can modify their software to load and edit Writeable Web pages. For example, Macromedia could modify Fireworks to load your Writeable Web purchase order, or Microsoft could add filters to Microsoft Office to load or save your Writeable Web college essay.



The older World Wide Web can become the Writeable Web if it wants to

Every feature explained above is actually implemented as a discrete open standard that existing web-sites can implement if they choose. For example, an older World Wide Web site such as www.codinginparadise.org could choose to provide an index and a search function by using XML in various places, without supporting the peer to peer aspect. This piecemeal, open approach means that all of the web can eventually move forward and adopt the advanced technologies of the Writeable Web without abandoning their existing investments.





  1. Non-Goals

Sometimes what you are trying not to achieve is as important as what you are trying to achieve. These are the non-goals of the version 1.0 release:

  • Perfection

    • Why: It's impossible.

    • Future: We'll get perfect eventually. :)

  • Development for anything more than Microsoft Windows.

    • Why: We just don't have the resources at this time. 1.0 and the first few point-releases will be Windows-specific.

    • Future: Every technology we use is aggresively cross-platform. Porting should not be hard; we just need the volunteers.

  • Microsoft Office compatibility

    • Why: Since we have 'rewritten' the underlying model of how you create documents with OpenOffice (i.e. abandoning the regular page and using frames/containers instead, for example), when Microsoft Word files are loaded OpenOffice features that we have dropped may be called on.

    • Future: This is a good goal for future versions.

  • OpenOffice compatibility.

    • Why: We have dropped many features, such as OpenOffice Formulas and macros, that the full OpenOffice format will depend on. We are still OpenOffice XML, though, and future compatibility should come online as we support more OpenOffice features.

    • Future: This is a good goal for future versions.

  • Searching the full Writeable Web.

    • Why: This is really really hard to do in a fast, reliable, completely decentralized way. While searching a single Writeable Web site will be supported and is crucial, searching the entire Writeable Web will not be done for the 1.0 release.

    • Future: This is a good goal for future versions.

  • Including OpenOffice Formulas, Spreadsheets, Charts, Drawings, HTML editing, and Presentations.

    • Why: Its just too much work to overhaul the user-interfaces of these applications to make them easier to use, and to make them fully-Writeable Web aware.

    • Future: These will come online in future releases.

  • Instant messaging compatibility with third-party systems.

    • Why: While we may go with using the Jabber instant messaging system, this does not mean that we will support interoperation with third-party messaging systems such as non-Writeable Web Jabber servers, AOL Instant Messenger, MSN Messenger, etc. It's just too hard for this release.

    • Future: This is a good goal for future versions.

  • OpenOffice Form functionality

    • Why: There are significant unresolved UI questions for what a form means when you have no centralized server. Who are you submitting this form to? Where does it go? Is there a clear divisions between editing a template and filling out a form (maybe not)? Plus, OpenOffice has some bad bugs in terms of layout issues with its form elements.

    • Future: This is a good goal for future versions.

  • AutoText

    • Why: AutoText is annoying and slightly useless. Dropping this OpenOffice feature simplifies the UI and lets us focus on other features.

    • Future: This probably doesn't have a future.

  • OpenOffice Databases

    • Why: OpenOffice databases are a really cool feature; unfortunately, there are many unresolved questions on how to handle these databases in a decentralized system such as Writeable Web. We just dont have the time to solve these issues.

    • Future: Might be part of a future release if the issues can be resolved.

  • SSL and electronic commerce for Writeable Web sites.

    • Why: This release is focused on non-commercial, creative, and political uses. Further, what does SSL mean to a distributed web site? What does it mean to pass your credit card number around between anonymous peers? Potentially dangerous stuff, so its not here.

    • Future: Might be part of a future release if issues can be resolved.

  • See changes and comparing documents.

    • Why: This is a feature of OpenOffice that is being dropped. It's a great feature, but it is a non-critical feature that needs to have its UI re-worked to deal with the collaborative reality of the Writeable Web.

    • Future: This is a good goal for future versions.

  • Inserting objects, OLE objects, plug-ins, and Java applets.

    • Why: There are significant security concerns with OLE objects. We are dropping plug-ins and Java applets to focus resources on other areas.

    • Future: Plug-ins and Java applets will probably be supported first in future versions. If security issues can be solved for OLE objects (which they might not), then they won't be supported.

  • Inserting or editing image maps.

    • Why: Killing this feature to focus time on other areas. This is a whole sub-UI we don't have time to focus on.

    • Future: Good feature for the future.

  • 3D effects

    • Why: 3D effects are OpenOffice feature-itis. Do we really need this here? I don't think so.

    • Future: Might have a future, probably with more of a task oriented UI (“Create a 3D newsletter banner!”) rather than a tool oriented one (“3D Stretch Tool”).

  • Anchoring “to character” and the Counter Editor.

    • Why: This feature lets you anchor an object to a specific character, and the counter editor lets you set up text flows around this anchored object. The UI is complex and nasty, and is probably used by 0.5% of the population.

    • Future: Probably has no future unless some compelling, main-stream use-cases can be shown.

  • Remote sites requesting more than username/public key from a user's personal passport.

    • Why: This is all that is needed for now.

    • Future: A user's personal passport will expand to include all sorts of personal information, stored under their control on their own personal machine. Remote sites can request information, such as the user's credit card number, and the user can decide whether to give this info to the remote site. Definently a future version.

  • Headers/Footers and direct footnotes.

    • Why: What do these mean when you have an endlessly long page?

    • Future: Who knows? Maybe they make sense for endlessly long web pages, or maybe the feature can be adoped to add a consistent header or footer across all pages of a writeable web site.

  • Editing non-writeable web sites or HTML. Further, Writeable Web is trying to go beyond the old HTML-web; by forcing the editor to do HTML as well, we will inherit flaws in that paradigm.

    • Why: Too much for this release.

    • Future: Far in the future, after Writeable Web has solidified its identity has a post-HTML editor so it doesn't become watered down by the well-developed HTML editing paradigm. Jump completely through the paradigm before reaching back to bring it forward.

  • Macros

    • Why: Security security security security.

    • Future: Will be included if a very strong security model is in place.

  • Configurable user-interface, such as making your toolbar be small icons or adding and removing individual icons.

    • Why: This is chrome that we just don't have the time to support. Plus it adds all sorts of nasty flexibility to the code that makes programming hard.

    • Future: I'm not terribly sympathetic to this need, but it will probably be included several versions down the line.

  • OpenOffice has a drawing feature where you can 'draw' a text box.

    • Why: This has been replaced with generalized OpenOffice frames. Plus, these text boxes are very limited and can't hold very much.

    • Future: No future; maybe for compatibility with OpenOffice.

  • OpenOffice Sections and Floating Frames

    • Why:  These are wierd, and provide too many ways of doing the same thing. Our One True Way(tm) is the Frame now, called the Container in Writeable Web.

    • Future: No future; maybe for compatibility with OpenOffice.

  • Gallery

    • Why: We just don't need it this release. Being killed to focus on other areas.

    • Future: Probably a future after being rethought a bit to make it peer to peer Internet intelligent (example: imagine a gallery that is auto-generated from the gallery resources of all peers?)

  • Line numbering

    • Why: Buggy feature of OpenOffice. Being killed to focus on other areas.

    • Future: Probably a future, but low priority.

  • Mozilla's Image Manager.

    • Why: This is a great idea that is too low level. It lets you block images from selected servers (like DoubleClick, for example). Unfortunately it is just way too low level. Why not instead bundle with the program the option “Turn off Ads”, and this will be auto-configured to a list of three-hundred ad servers, for example? The UI needs to be re-thought.

    • Future: Has a future, though low priority. Needs a better user-interface.

  • Advanced control over how Mozilla stores its non-Writeable Web site password cache

    • Why: Mozilla didn't think this UI through too well; it suffers from featuritis when it should simply be invisible. Mozilla by default obfuscates the password cache, giving the option of encrypting the cache instead. We simply take the user's Writeable Web password and use this to encrypt the Mozilla password cache (if there are bugs with Mozilla's password encryption system then obfuscation will be used instead).

    • Future: No future; we do it in a way where option-itis isn't needed.

  • Having anything other than an Online View.

    • Why: Writeable Web isn't a word processor for print; its a word processor for internet communities. I don't want to hold it back by forcing print features in at this time; it needs to develop its own vision first.

    • Future: Future versions of Writeable Web might make it easier to do print work, but only after it has achieved a distinct Internet identity first. Otherwise it will be watered down by the more developed print word processing paradigm.

  • Bundling an email, calendering, or IRC program. Mozilla's XUL apps for these will be dropped.

    • Why: We can't bite off more than we can chew right now.

    • Future: These will probably be included in Writeable Web in the future. Maybe not to avoid bloat, which is something Writeable Web needs to be careful of since it tries to be an integrative program.

  • True anonymity.

    • Why: This is a major task unto itself. The UI is already there for this (i.e. you can choose Anonymous when you login), but this anonymity simply means that you can log into Writeable Web sites without using a public key. Your IP address is still visible, and you peer directly requests contents from other peers and is not “proxied” or hidden in any way. Also, true anonymity imposes heavy performance costs on the network.

    • Future: Anonymity is a valuable goal that future contributors can add. The nice thing is the UI will not have to change to add true anonymity.

  • Encrypted private communication.

    • Why: Another major task that is not a goal of Writeable Web at this time. This feature would also impose large performance costs.

    • Future: This will probably not be a major goal for the next few releases, unless small amounts of information are being encrypted (such as commercial transactions). Maybe if every peer is also made into an SSL server for future releases, we'll get this feature for “free”.

  • Supporting Mozilla's sidebar entries such as Search, Weather, New Mozilla Builds, etc.

    • Why: The idea behind these bars slightly conflicts with Writable Web's existing use of a sidebar. They also add too many entries to the sidebar.

    • Future: Might be supported if the UI issues can be ironed out.

  • Digital signatures exposed to the user.

    • Why: While there is an Insert > Signature option in the pull-down menus, this is simply a standardized textual signature and not a digial one.

    • Future: This feature is important, and will be added in the future. The nice thing is the UI will not change since the digital information will simply underlie the Insert > Signature already there. However, the UI will need to add something to the side of the signature to say “Signature Valid” or “Signature Not Valid.”

  • Mozilla Skins.

    • Why: This is a non-core feature that is not important for this release.

    • Future: This is an important feature for the future; compatibility with Mozilla skins is planned.

  • Parodies

    • Why: It turns out this is pretty hard to do for old HTML oriented sites. First, we need to grab the whole site and provide controls for the user when they are dealing with a huge site. Second, we need to convert this into Writeable Web format since we don't support HTML editing.

    • Future: As soon as the OpenOffice HTML editor is ported to Writeable Web, this feature will be supported. It is considered an extremely important feature and will come online soon.

  • Categorization of comments in the comment areas or finding commented What's Related entries or comments using .comment. Older conceptions of Writeable Web included a concept known as .comment endings. Any site, such as www.nike.com, could have a .comment added to the end followed by a subject, such as www.nike.com.comment.LaborPolicy . When the user followed this link, an autogenerated site controlled by no one would be presented culled from comments and What's Related entries.

    • Why: The UI for this was complex, and it was questionable if people would look for subjects using the .comment system instead of simply using the What's Related tab. Dropped to gain simplicity. Also, would user's manually type their comment using a pull-down? It is questionable.

    • Future: Probably none.

  • Any new technology, development, or function point not enumerated in this document.

    • Why: Features have to be pinned down, even if they aren't perfect. Feature-creep is extremely dangerous.

    • Future: If a promising new technology or function comes along that is worthwhile, it should be written down so we can incorporate it into future versions.

  • Hard-core application performance, as opposed to network performance.

    • Why: The focus of this release is functionality and optimizing network performance as opposed to client application performance.

    • Future: This is a very important requirement, and will probably come in the 1.1 or 1.2 releases.

  • Message moderation. This would mean adding a distributed Slashdot-style moderation system to the message boards on each site.

    • Why: This is hard to do from a UI perspective in such a distributed fashion. There are other unresolved issues as well. It's just too much to bite off for this release.

    • Future: This is a definent feature for a post 1.0 release.

  • Massive collaborative filtering. This is a decentralized way to avoid abusing the system, as well as keeping the noise level down and the signal level up. For example, random user's would be asked to identify gibberish domain names, and their responses would be automatically blended together to remove domain names that are simply there to abuse the system.

    • Why: This is a hard thing to finish both from a UI perspective and an architectural one. Plus, I don't want to over-engineer moderation/filtering until the core system itself isn't solid; it doesn't make sense to solve a problem that isn't a problem yet. I want to see how the problem itself shapes up first.

    • Future: Definently has a future.

  • Being able to change your comments once you've posted them on a message board.

    • Why: Not a core requirement for this release.

    • Future: Maybe in the future.

  • Internationalization of the client user-interface.

    • Why: Too much to bite off for now. Plus, Mozilla and OpenOffice have different underlying internationalization architectures. However, it is an explicit goal of this release to have a fully internationalized network, such that domain names can be internationalized, file formats are internationalized, etc. However, this internationalization will not be exposed to the user in this release (i.e. you can type in a Japanese domain name in the address bar yet).

    • Future: This is an extremely important feature for a soon after 1.0 release.

  • OpenOffice Master Documents

    • Why: The use of these is questionable when we have AutoNames.

    • Future: Probably none, unless I don't understand their function correctly (which I suspect)

  • Autosuggestion of KnowledgeFields values in AutoTemplates based on previous values for this field type.

    • Why: This is a great feature, but it is non-core and can wait for the future. Plus, there are probably memory issues with holding around all of the previous values for all of the different field types.

    • Future: Definently a feature for the future.

    • Why: While Writeable Web is built above cross-platform technologies (OpenOffice, Mozilla, XUL, Java, etc.), it is simply not feasible for the small number of current developers to build, test, and deploy Writeable Web across multi-platforms at this time.

  • Hard-drive requirements not yet set (will probably be pretty high)

    • Why: While the Writeable Web is free as in money, it is not free as in resources; your payment for using and creating within the Writeable Web are bandwidth and hard-drive resources.
