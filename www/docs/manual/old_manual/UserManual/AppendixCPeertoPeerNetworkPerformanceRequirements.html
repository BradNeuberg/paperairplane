Appendix C - Peer-to-Peer Network Performance Requirements

The following are the performance characteristics that the peer to peer Two Way Web network should elicit. Some operations may never complete and should report timeouts.

  • Sign In time - 10 seconds in extreme case, 3 seconds in normal case

  • Sign Out time - 10 seconds in extreme case, 2 seconds in normal case

  • Pulling up main Two Way Web Directory or sub-category list - should be near instataneous, since no network communication is involved

  • Pulling up Two Way Web Directory sub-category - 2 seconds best case, 3 seconds normal case, 10 seconds worst case

  • Pulling up a Two Way Web page of 50K you've never visited before - 5 seconds or less

  • Pulling up a Two Way Web page that is in your cache - should feel instataneous

  • Saving a Two Way Web Page of 50K into the network - 2 seconds in best case, 3.5 seconds in normal case, 10 seconds in worst case. Should instantly return after propagation starts.

  • Creating a New Two Way Web Site - 5 seconds normal case, 10 seconds worst case

  • Joining a Site - 5 seconds normal case, 10 seconds worst case

  • Propagating changes to a Two Way Web Page - Others should start to see them right when it is saved, with no delay.

  • Deleting a Two Way Web Page - 5 seconds normal case, 10 seconds worst case.

  • Loading a site's metadata for AutoLinking features - 1 second in best case, 8 seconds worst case

  • Length of time a new Two Way Web Site becomes visible - 5 minutes best case, 15 minutes normal case, 30 minutes to an hour worst case.

  • Length of time it takes until a change to a site's structure shows up for AutoLinking feature - define

Sites should show linear or near-linear performance versus size of site and number of participants, i.e. everything should be log n or n log n in terms of performance, where n could be considered either the number of nodes in the system or the amount of content for that particular function, such as number of domain endings.

Further metrics of performance, such as avalability, reliability, latency, minimizing the burdens of individual peers, etc. are defined in the Two Way Web Architectural Document [link me].
